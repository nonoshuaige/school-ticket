## 一、项目概述

### 1.1 项目背景

面向毕业晚会、歌手大赛等校内热门活动的票务开售场景，构建一套从前端售票、订单创建、支付关单、取消回补到退款补偿的订单履约闭环系统。v3.0 新增笔记社区（发笔记、点赞、关注）。v3.1 新增发布笔记与二级展平评论系统。v3.2 引入 Redis ZSET 游标分页与 RabbitMQ 异步消息，实现笔记排行（最热/最新/我的）与关注/粉丝列表高性能查询。v3.3 重构笔记社区为推荐流（布隆消重）+ 关注流（拉模式时序聚合）双 Feed 架构，个人中心收敛"我的笔记"入口。v3.4 关注流升级为推模式收件箱（fanout-on-write），新增 VO 缓存层（note:vo Hash）与 Redis 先行点赞计数器，认证体系升级为 Cookie + Authorization Header 双通道。v3.5 新增用户点赞 Set（user:likes:{userId}）实现 O(1) isLiked 查询，4 处 Redis Pipeline 批量优化将推荐流延迟从 600ms 压至 30ms。v3.6 重构 Redis 缓存架构为按用户懒加载 + 全 Key TTL 过期策略，冷启动仅同步全局池，用户级数据在首次访问时从 MySQL 按需重建，不活跃用户不占内存。v3.7 重构抢购链路：Caffeine 本地售罄缓存 → Snowflake 预生成订单号 → Redis Lua 原子扣库存 + 跨票档限购 + Stream 写入 → Stream Consumer 异步落库 MySQL，取消/退款/超时 Lua 回滚库存；购买追踪 Key 从用户层（user:purchase:{userId}:{eventId}）下沉至票档层（ticket:purchase:{ticketId}）。v3.8 重构逆向链路可靠性：取消/超时关单引入本地消息表 `order_event_log`（与 MySQL 同事务写入），定时任务异步驱动 Redis Lua 回滚，重试上限 5 次；退款链路引入退款表 `refund`（refund_id=order_no），退款请求快速入表返回，RefundTask 异步消费执行实际退款。v3.9 活动列表改为 Redis 优先 + 前端热卖/预热 Tab 分开展示 + 读时修复脏数据。v3.10 新增前端幂等保护：UUID 幂等键 + Redis SET NX 原子抢占 + DuplicateKeyException 兜底，防止重复提交/网络重试产生重复订单。v3.11 关注流从纯推模式升级为推拉结合：普通用户（<1000粉）继续 fanout 到粉丝收件箱，大V（≥1000粉）发帖时仅写 `note:mine` 不 fanout，粉丝读关注流时合并收件箱（推内容）+ 大V 发件箱（拉内容）按时间戳排序。

## 三、系统架构

### 3.1 认证流程

1. 登录成功 → 后端将 JWT 写入 httpOnly Cookie（`Set-Cookie: token=...; Path=/; HttpOnly; SameSite=Lax`），同时响应体返回 token
2. 前端将 token 存入 `localStorage.authToken`，请求拦截器自动附加 `Authorization: Bearer <token>` 头
3. JWT 过滤器拦截请求 → 优先解析 Cookie，Cookie 为空时兜底解析 Authorization header → 提取 userId → 写入 CurrentUserHolder
4. 业务层通过 CurrentUserHolder.getUserId() 获取当前用户
5. 请求结束 → CurrentUserHolder.remove() 清除 ThreadLocal
6. Spring Security 未认证请求返回 HTTP 401（非默认 403），前端 401/403 统一清理 token 并跳转登录页

> **双通道认证**：Cookie（SameSite=Lax，无 Domain）为主要通道，Authorization Header 为兜底。解决浏览器对 `localhost` 无嵌入点域名拒绝 `Domain=localhost` Cookie 的兼容性问题。

### 3.2 CurrentUserHolder

```java
public class CurrentUserHolder {
    private static final ThreadLocal<Long> USER_ID_HOLDER = new ThreadLocal<>();
    public static void setUserId(Long userId) { USER_ID_HOLDER.set(userId); }
    public static Long getUserId() { return USER_ID_HOLDER.get(); }
    public static void remove() { USER_ID_HOLDER.remove(); }
}
```

> 约定：禁止在 Controller 参数中传递 userId。统一从 CurrentUserHolder 获取。

### 3.3 笔记模块认证

笔记列表接口为公开访问（`permitAll()`），但点赞和关注操作需要登录。Controller 层使用 `tryGetUserId()` 和 `requireUserId()` 两个辅助方法区分：
- `tryGetUserId()`：静默获取，未登录返回 null，列表仍正常返回但不带当前用户状态
- `requireUserId()`：强制获取，未登录抛出 `BusinessException(401, "请先登录")`

### 3.4 Redis 数据结构与 Feed 流架构

> v3.6 重构：所有 Key 统一增加 TTL 过期时间，用户级数据改为首次访问时懒加载，冷启动仅同步全局池。不活跃用户不占内存。

**TTL 总览**：

| TTL | 适用范围 |
|-----|---------|
| 1天 | `note:latest`、`note:hottest`（全局池，可从 DB 重建） |
| 3天 | `note:mine:{userId}`、`user:follow:{userId}`、`user:fans:{userId}`、`user:likes:{userId}`、`feed:following:{userId}`（用户级，按需懒加载） |
| 无（持久） | `bigv:ids`（大V 标记 Set，粉丝数跨阈值时增删） |
| 7天 | `note:vo:{noteId}`、`note:like:count:{noteId}`、`user:bloom:seen:{userId}` |
| 30分钟 | `feed:recommend:{userId}`（会话级快照） |
| saleEndTime | `ticket:stock:{ticketId}`（票档库存预热） |
| 5分钟 | `ticket:soldout:{ticketId}`（售罄标记） |
| 无（事件维度） | `ticket:purchase:{ticketId}`（票档层购买记录 Hash，userId→qty） |
| MAXLEN ~10000 | `stream:orders`（订单事件 Stream） |

**活动缓存**（v3.9 新增）：

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `event:pool:hot` | ZSET | saleEndTime | 热卖中活动 ID，score=eventStartTime，启动时从 status=1 活动预热 |
| `event:pool:warmup` | ZSET | max(saleStartTime)+1d | 预热中活动 ID，启动时从 status=0 活动预热 |
| `event:vo:{eventId}` | String (JSON) | saleEndTime | 活动完整信息（含预计算 minPrice），Jackson 序列化 |
| `event:tickets:{eventId}` | String (JSON) | saleEndTime | 票档列表 JSON 数组 |

活动查询链路（v3.9）：
- 前端按 Tab 分调 `GET /event/list?status=1` 或 `status=0`
- 后端 Redis ZSET 分页 → HGETALL VO → 直接返回（不碰 MySQL）
- VO 缓存缺失时从 MySQL 回填并写入 Redis
- **读时修复**：取出的 event 若 status 与 pool 预期不符（DB 状态已变更），自动从 pool 移除脏数据

**推荐流**（`GET /api/v1/note/recommend-feed`）—— 快照拉取 + 布隆过滤器曝光消重：

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `note:latest` | ZSET (score=创建时间) | 1天 | 全站最新候选池（快照生成数据源），过期从 DB 重建 |
| `note:hottest` | ZSET (score=点赞数) | 1天 | 笔记热度排行 |
| `user:bloom:seen:{userId}` | String (Bitmap) | 7天 | 布隆过滤器，~12KB/人 |
| `feed:recommend:{userId}` | List | 30分钟 | Session 级私有 Feed 快照队列 |

推荐流链路（v3.3+）：
- 下拉刷新（cursor=null）：`note:latest` 取最新 600 条 → shuffle → 布隆过滤 → 截取 200 条 → `RPUSH feed:recommend:{userId}` → `LPOP pageSize` 条返回
- 上拉加载（cursor≠null）：私有 List `LPOP pageSize` 条返回，不再依赖全局 ZSET Offset，彻底解决数据错位
- 懒加载：`note:latest` 过期自动从 `user_note` 全表重建

**关注流**（`GET /api/v1/note/following-feed`）—— 推拉结合（v3.11 升级）：

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `feed:following:{userId}` | ZSET (score=发布时间) | 3天 | 粉丝收件箱，仅普通用户发布时 fanout 写入，最多保留 800 条 |
| `bigv:ids` | Set | 无（持久） | 大V 用户 ID 集合（粉丝数 ≥ 阈值），读关注流时用于筛分 |
| `user:follow:{userId}` | ZSET (score=关注时间) | 3天 | 关注作者列表，过期从 MySQL 懒加载 |
| `user:fans:{userId}` | ZSET (score=关注时间) | 3天 | 粉丝列表（fanout 时查询），过期从 MySQL 懒加载 |
| `note:mine:{authorId}` | ZSET (score=创建时间) | 3天 | 各作者发布的笔记索引（大V 拉模式 + 新粉回填数据源） |

关注流链路（v3.11 推拉结合）：
- **发布时分支**：检查作者粉丝数 → 粉丝 < 阈值（1000，`feed.big-v-threshold`）→ 推模式，fanout 到粉丝收件箱；粉丝 ≥ 阈值 → 拉模式，仅写 `note:mine:{authorId}` 不 fanout，标记 `bigv:ids`
- **读取时合并**：区分关注列表中的大V（`bigv:ids` ∩ followingIds）和普通关注者 → 收件箱（普通用户推内容）ZREVRANGEBYSCORE 游标分页 + 并行拉取各 `note:mine:{bigVId}` → 两个来源的 (noteId, score) 合并 → 按时间戳降序 → 取 pageSize 条 → VO 组装
- **新粉关注**：从 `note:mine:{authorId}` 拉取被关注者最近 50 条笔记，回填到新粉丝收件箱（大V 和普通用户同一逻辑）
- **取关**：遍历 `note:mine:{authorId}` 从收件箱 ZREM 清除该作者全部笔记 + 检查粉丝数是否跌破阈值，摘除大V 标记
- **删笔记**：普通用户遍历粉丝列表 ZREM，大V 仅清除 mine + latest ZSET
- **配置项**：`feed.big-v-threshold: 1000`（application.yml），可按需调整阈值

**VO 缓存层**（v3.4 新增，v3.6 改为按需写入）：

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `note:vo:{noteId}` | Hash | 7天 | 笔记 VO 全量缓存（content, userId, nickname, phone, createTime, likeCount） |

- 发布笔记时写入 VO 缓存并设置 TTL
- 推荐流/关注流列表优先从 VO 缓存批量读取，miss 查 MySQL 并回填（读时修复）
- 点赞/取消点赞通过 `HINCRBY` 增量更新 VO 中的 likeCount 并续期 TTL
- v3.6 起冷启动不再全量预加载 VO，改为按需写入——只有被实际访问到的笔记 VO 才进入 Redis

**点赞计数器**（v3.4 新增，Redis 先行）：

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `note:like:count:{noteId}` | String | 7天 | 点赞计数（INCR/DECR），异常时自动修复 |
| `user:likes:{userId}` | Set | 3天 | 用户点赞过的所有 noteId，O(1) SISMEMBER 判赞 |

- 点赞：`INCR` + `SADD user:likes:{userId} noteId` → 写 MySQL → `ZADD note:hottest` → `HINCRBY note:vo:{noteId} likeCount 1`，每次操作续期 TTL
- 取消点赞：`DECR` + `SREM user:likes:{userId} noteId` → 删 MySQL → MySQL 实际值校准 Redis 计数器 → 更新 hottest ZSET
- 判断 isLiked：`SISMEMBER user:likes:{userId} noteId`（批量用 pipeline），首次访问时从 MySQL note_like 表懒加载
- v3.6 起冷启动不再全量加载用户点赞 Set，改为访问时按用户懒加载

**Pipeline 批量优化**（v3.5 新增）：

推荐流首次加载原需 ~350 次 Redis 往返（布隆逐条 GETBIT ×200 + 逐条 SETBIT ×80 + VO 逐条 HGETALL ×10 + 逐条 SISMEMBER ×10），v3.5 将 4 处全部 pipeline 化，合并为 4 次往返：

| 优化点 | 之前 | 之后 |
|--------|------|------|
| 布隆检查 200 候选 | 200 次 GETBIT | 1 次 pipeline |
| 布隆标记 10 条 | 80 次 SETBIT | 1 次 pipeline |
| VO 缓存批量读 | 10 次 HGETALL | 1 次 pipeline |
| isLiked 批量查 | 10 次 SISMEMBER | 1 次 pipeline |
| 快照弹出 | 10 次 LPOP | 1 次 pipeline |
| 空白布隆短路 | 无 | EXISTS 跳过，节省 130ms |

> 推荐流冷启动 ~420ms（含首次 Lettuce 连接 ~220ms），热路径降至 **23-30ms**。

**抢购链路**（v3.7 重构，v3.10 幂等保护）：

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `idempotent:order:{key}` | String | 300s | 幂等键（前端 UUID），SET NX 原子抢占，值="PENDING"→orderNo |
| `ticket:stock:{ticketId}` | String | saleEndTime | 预热写入的票档剩余库存，Lua DECRBY 扣减 |
| `ticket:soldout:{ticketId}` | String | 5分钟 | 售罄标记（"1"=售罄），Caffeine 本地缓存短路 |
| `ticket:purchase:{ticketId}` | Hash | 无 | 票档层购买记录 `{userId: quantity}`，每票档每人最多 5 张 |
| `stream:orders` | Stream | MAXLEN ~10000 | 订单事件，消费者异步落库 MySQL，幂等主键去重 |

抢购链路流程：
```
POST /api/v1/order/create { ticketId, quantity, idempotencyKey }
  │
  ├─ 0. 幂等保护：Redis GET idempotent:order:{key}
  │      ├─ =orderNo → 查 MySQL 返回已有订单（幂等命中）
  │      ├─ ="PENDING" → 409 "请求处理中"
  │      └─ =null → SET NX EX 300 原子抢占 → 抢占失败同上
  │
  ├─ 1. Caffeine.get(ticketId) → soldOut? → 直接返回"已售罄"
  ├─ 2. 无锁 selectById 查票档信息 + 活动信息
  ├─ 3. IdUtil.getSnowflakeNextIdStr() 预生成订单号
  ├─ 4. 查活动下所有票档 ID（跨票档检查用）
  ├─ 5. 执行 purchase.lua（原子操作）:
  │     KEYS: stock, soldout, purchase, stream
  │     ARGV: orderId, userId, ticketId, qty, maxPerUser(5), totalStock, totalPrice, expireTimeMs, [allTicketIds...]
  │     ├─ soldout=1? → 返回 -1
  │     ├─ stock < qty → SET soldout=1 → 返回 -2
  │     ├─ 跨票档检查：遍历其他 ticket:purchase:{otherId}，HGET userId > 0? → 返回 -4
  │     ├─ HGET 本票档 userId + qty > 5? → 返回 -3
  │     └─ OK → DECRBY stock, HINCRBY purchase, XADD stream → 返回 0
  │
  ├─ 6. Lua 返回 -1/-2 → Caffeine.put(ticketId, true) → releaseIdempotentKey → 返回"售罄"
  │     Lua 返回 -3 → releaseIdempotentKey → 返回"超出购买限制"
  │     Lua 返回 -4 → releaseIdempotentKey → 返回"跨票档冲突"
  │
  ├─ 7. 直接 INSERT MySQL（前端立即获取订单）
  │      ├─ 成功 → SET idempotent:order:{key} = orderNo
  │      └─ DuplicateKeyException → Stream 已先落库 → 幂等返回已有订单
  │
  └─ 8. Stream Consumer 异步幂等兜底落库
```

**回滚链路**（取消/超时关单 → 本地消息表驱动，v3.8 重构）：
```
cancel / expire
  │
  ├─ 1. MySQL 事务内: UPDATE order status=2, UPDATE ticket_category remaining+=qty
  │         └─ INSERT order_event_log (status=0, eventType=1或3)
  ├─ 2. Caffeine invalidate（立即释放售罄标记，不等 Redis 回滚）
  └─ 3. OrderEventLogTask 每 10s 扫描 status=0
       ├─ 执行 rollback.lua:
       │    KEYS: stock, soldout, purchase  ARGV: userId, quantity
       │    ├─ INCRBY stock → DEL soldout
       │    └─ HGET purchase userId → HINCRBY -qty 或 HDEL
       ├─ 成功 → UPDATE status=1（已确认）
       └─ 失败 → retry_count+1，满 5 次 → status=2
```

**退款链路**（退款表异步消费，v3.8 重构）：
```
refund
  │
  ├─ 1. POST /api/v1/order/refund → INSERT refund (refund_id=orderNo, status=0) → 立即返回
  ├─ 2. RefundTask 每 10s 扫描 refund WHERE status=0
  │      └─ processRefund: 回补名额 + order status=3 + INSERT order_event_log (eventType=2)
  │           + Caffeine invalidate → refund status=1/2
  └─ 3. OrderEventLogTask 消费 event_log → rollback.lua → ack
```

**Caffeine 本地售罄缓存**：
- `Cache<Long, Boolean>`（ticketId → isSoldOut），expireAfterWrite=5s，maximumSize=1000
- 仅缓存售罄=true，库存恢复时（取消/退款/补货）主动 invalidate
- 超卖的最后一道防线仍由 Lua 原子操作保证

**我的笔记**（`GET /api/v1/note/my-notes`）：

| Key | 类型 | 用途 |
|-----|------|------|
| `note:mine:{userId}` | ZSET (score=创建时间) | 当前用户发布的笔记索引 |

**关注/粉丝列表**：

| Key | 类型 | 用途 |
|-----|------|------|
| `user:follow:{userId}` | ZSET (score=关注时间) | 我关注的人 |
| `user:fans:{userId}` | ZSET (score=关注时间) | 我的粉丝 |

**双写策略**：MySQL 写入成功后同步更新 Redis ZSET，同时发送 RabbitMQ 消息作为异步兜底。

### 3.5 RabbitMQ 异步消息

Topic Exchange `school.ticket.exchange` 绑定 4 个队列：

| Queue | Routing Key | 用途 |
|-------|------------|------|
| `note.like.queue` | `note.like` | 点赞/取消后异步更新 Redis hottest ZSET |
| `note.create.queue` | `note.create` | 发布笔记后异步写入 latest + mine ZSET + 推拉结合 fanout（大V 跳过） |
| `note.delete.queue` | `note.delete` | 删除笔记后异步清除 Redis ZSET + VO 缓存 |
| `user.follow.queue` | `user.follow` | 关注/取关后异步更新 follow/fan ZSET + 收件箱回填/清理 |

> `RedisSyncListener.java` 中 4 个 `@RabbitListener` 方法消费消息，执行与同步写入相同的 ZSET 操作。
>
> **软删除一致性保障**：
> - `user_note` 表新增 `is_deleted` 字段（@TableLogic），删除操作软标记 + 同步 ZREM + MQ 异步兜底
> - 推荐流/关注流 VO 组装时进行读时修复（Read-Repair）：通过 `selectBatchIds` 发现 @TableLogic 过滤掉的已删除 ID，主动从 Redis 全局 ZSET 中 `ZREM` 清除，防止脏 ID 持续分发

### 3.6 冷启动同步

系统启动后需调用同步接口，将 MySQL 全局池数据写入 Redis（用户级数据由懒加载按需处理）：

```
POST /api/v1/note/sync-to-redis        # 同步 note:latest + note:hottest 全局池 + 点赞计数器（无需登录）
POST /api/v1/user/follow/sync-to-redis # 同步当前登录用户的关注关系（需登录，仅同步该用户）
```

> v3.6 起冷启动仅同步全局池。用户级数据（关注/粉丝/收件箱/点赞/我的笔记）在用户首次访问对应功能时从 MySQL 懒加载，并自动设置 TTL。不活跃用户不消耗 Redis 内存。

## 五、订单状态机与履约流程

### 5.1 状态流转

```
[*] → 待支付(0) → 已支付(1) → 已核销(4)
待支付(0) → 已取消(2)  [主动取消/超时关单]
已支付(1) → 已退款(3)  [申请退款]
```

### 5.2 关键规则

**下单（幂等保护 + Lua 原子扣库存 + Stream 异步落库，v3.7/v3.10）**
- 前端生成 UUID 幂等键 → Redis SET NX 原子抢占幂等键（抢占失败 409）→ Caffeine 本地售罄缓存短路（5s TTL）→ 无锁 selectById 查票档/活动 → Snowflake 预生成订单号 → Redis Lua 原子执行（售罄检查 + 库存扣减 + 跨票档限购 + 每人每票档≤5 + Stream 写入）→ 直插 MySQL → 成功则幂等键指向订单号
- DuplicateKeyException 时 Stream 已先落库 → 查已有订单幂等返回
- 订单 status=0, expire_time=NOW()+15min，Lua 内部通过 ticket:purchase:{ticketId} Hash 记录每人每票档购买量

**支付**（模拟）
- 验证订单存在且 status=0 且未过期 → 置 status=1, paid_time=NOW()

**取消/超时关单（本地消息表驱动 Redis 回滚，v3.8 重构）**
- 主动取消：POST /api/v1/order/cancel → MySQL 事务内：回补名额 + UPDATE status=2 + INSERT order_event_log (status=0) → Caffeine invalidate
- 超时关单：@Scheduled(fixedRate=30s) 扫描 expire_time < NOW() AND status=0 → 同上流程（回补 + 改状态 + 写 event_log）
- OrderEventLogTask：@Scheduled(fixedRate=10s) 扫描 order_event_log WHERE status=0 → 执行 rollback.lua（INCRBY stock + DEL soldout + DEL/HINCRBY purchase）→ 成功 ack status=1，失败 retry_count+1，满 5 次 → status=2
- 本地消息表与 MySQL 状态变更为同一事务，保证原子性；Redis 临时不可用时正常返回，由定时任务补偿执行

**退款（退款表异步消费，v3.8 重构）**
- 退款申请：POST /api/v1/order/refund → INSERT refund (refund_id=orderNo, status=0) → 立即返回
- RefundTask：@Scheduled(fixedRate=10s) 扫描 refund WHERE status=0 → 执行实际退款：回补名额 + order status=3 + INSERT order_event_log（驱动 Redis 回滚）+ Caffeine invalidate → 成功 ack status=1，失败 status=2
- OrderEventLogTask 消费 event_log（eventType=2 退款）→ rollback.lua → ack

## 九、运行指南

### 9.1 启动 MySQL

MySQL 安装路径：`D:\JavaDevelop\mysql-8.0.28-winx64\`，data 目录已初始化，`my.ini` 已配置。

**启动方式**（Bash 终端）：
```bash
# 1. 先清理残留进程（如果有）
taskkill //f //im mysqld.exe 2>/dev/null

# 2. 后台启动 MySQL
"D:/JavaDevelop/mysql-8.0.28-winx64/bin/mysqld.exe" --defaults-file="D:/JavaDevelop/mysql-8.0.28-winx64/my.ini" &

# 3. 等待启动完成
sleep 3
```

> **my.ini 内容**：`basedir`、`datadir`、`port=3306`、`character-set-server=utf8mb4`。mysqld 会读取该文件，无需每次手动传 `--datadir` 和 `--port` 参数。
>
> **常见问题**：
> - 启动报 `ibdata1 must be writable`：检查 data 目录权限，或用管理员终端运行
> - 端口 3306 被占用：`tasklist | grep mysqld` 检查旧进程，`taskkill //f //im mysqld.exe` 强杀后重试
> - `my.ini` 路径报错：确保文件编码为 ANSI/UTF-8 without BOM，路径用正斜杠 `/` 或双反斜杠 `\\`
> - Docker Desktop 未启动时 mysqld 也会受影响的可能较小，但建议先确认 Docker 已在运行

**验证**：
```bash
# 检查进程是否存在
tasklist 2>/dev/null | grep -i mysqld

# 测试连接
"D:/JavaDevelop/mysql-8.0.28-winx64/bin/mysql.exe" -u root -p123456 -e "SELECT 1"
```

## 附 A：开发问题记录

| 问题 | 解决 |
|------|------|
| 下单 403 | JWT Cookie Filter 补上 SecurityContextHolder.setAuthentication() |
| MySQL 8.0.36 CDN 不可达 | 改用阿里云镜像 8.0.28 |
| JDBC charset=utf8mb4 报错 | 改为 characterEncoding=UTF-8 |
| 前端无限跳回首页 | Pinia storeToRefs() 解构获取响应式 ref |
| 订单列表未区分状态 | 加 Tab 筛选 + 后端分页+状态筛选 |
| 未开售可购买 | isOnSale 计算属性控制票档禁用 |
| mysqld 启动报 ibdata1 must be writable | 检查 data 目录权限，用管理员终端运行；或检查 my.ini 文件是否存在 |
| mysqld 启动报 Fatal error in defaults handling | `my.ini` 文件不存在，在 `D:\JavaDevelop\mysql-8.0.28-winx64\` 下创建含 `[mysqld]`、`datadir`、`port` 的 ini 文件 |
| 每次启动 MySQL 都出问题 | 确认 `my.ini` 已创建 + 先 `taskkill //f //im mysqld.exe` 清理残留进程 + 用 bash 风格命令启动（见 9.1） |
| Redis note 数据过期/异常 | 执行 9.5 节第 0 步清除 `note:*` `feed:*` `user:bloom:*` 三个前缀的 key，再重新执行冷启动同步 |
| PowerShell `mysql < init.sql` 报 RedirectionNotSupported | `<` 是 PS 保留运算符，改用 `cmd /c` 包装 |
| init.sql 执行报 No database selected | CREATE DATABASE / USE 被注释，加 `-D school_ticket` |
| pom.xml 自引用依赖导致 Maven 构建失败 | 删除 `<dependency>com.schoolticket:school-ticket</dependency>` |
| 活动卡片价格硬编码 | Event entity 加 @TableField(exist=false) minPrice，Service 批量查 ticket_category 取 MIN(price) |
| 笔记 N+1 查询用户信息 | 使用 userMapper.selectBatchIds() 批量加载 |
| 笔记点赞 N+1 查询 | LambdaQueryWrapper.in(noteIds) 批量查 + 按 noteId 分组 |
| gen_data.js 覆盖测试账号 | user_id=1 显式插入 '13800138000' / '测试用户' |
| 关注列表/笔记/点赞未落库 | 建表 + entity + mapper + service + controller 全套后端 |
| 笔记排行榜切换无变化 | van-tabs v-model 绑定问题，改用自定义 span 按钮 + 显式 switchSort() |
| gen_data.js 生成未来日期 | 日期生成范围限制到 2026-06-21 之前 |
| 前端分页组件不适用于游标 | 替换 van-pagination 为"加载更多"按钮 + nextCursor 驱动 |
| Redis isLiked 永远为 false | RedisNoteRankingService 中 isLiked 检查从未写入的 Redis key，改为 MySQL 批量查询 note_like 表 |
| /user/follow/sync-to-redis 403 | 该接口在 `/api/v1/**` authenticated() 规则下，需带登录 Cookie |
| 关注流/推荐流 500 错误 | `batchGetNoteVOs` 使用低层级 `executePipelined` 与 StringRedisTemplate 序列化器不兼容，改为 `opsForHash().entries()` 逐 key 读取 |
| 点赞 500 "ERR value is not an integer" | `incrLikeCount`/`decrLikeCount` 遇到 Redis 中非整数值（旧同步残留），加 try-catch 删除 key 重建 |
| VO likeCount HINCRBY 报错 | `updateVOLikeCount` 遇到 Hash 中非整数 likeCount，加 try-catch 读取计算后 PUT 回写 |
| Cookie 跨端口不发送 | `Domain=localhost` 被浏览器拒绝（RFC 6265 要求嵌入点），改为无 Domain + SameSite=Lax + Spring ResponseCookie |
