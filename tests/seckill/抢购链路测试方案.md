# 抢购链路测试方案

本文档用于验证当前校园票务项目的抢购链路是否真正满足设计目标，而不是只验证“用了 Redis、Lua、RabbitMQ”。测试分为基线准备、并发压测、异常注入和最终一致性对账四类。

## 1. 测试目标

### P0 目标

1. **零超卖**：任意并发下，成功订单占用数量不能超过票档总库存，Redis 库存和 MySQL 库存不能为负。
2. **活动级限购有效**：同一用户同一活动跨多个票档累计最多 5 张，不能换票档绕过。
3. **无效流量快速失败**：售罄、限购、频控后的请求应在本地缓存或 Redis 层失败，不能持续打到 MySQL。
4. **异步成单可靠**：Lua 预扣成功后，即使 RabbitMQ/Consumer 短暂异常，也要通过 Redis Stream + PEL 重投最终落库，重复消息不产生重复订单。
5. **逆向补偿最终一致**：取消、超时关单等非支付逆向流程执行后，库存和限购记录最终回滚到正确值。

### 建议记录指标

- 成功订单数、失败请求数、业务失败原因分布。
- 响应延迟 p50/p95/p99。
- MySQL `order` 表订单数、`ticket_category.remaining_quantity`。
- Redis `ticket:stock:{ticketId}`、`event:purchase:{eventId}`。
- Redis Stream PEL 待确认数量。
- RabbitMQ 队列堆积数和死信数。

## 3. 压测脚本

仓库内提供了 Node 20 原生脚本：

```powershell
cd D:\MyProjrct
node .\tests\seckill\load-test.mjs --mode load
```

可配置环境变量：

```powershell
$env:BASE_URL="http://localhost:8080/api/v1"
$env:TICKET_ID="21"
$env:QUANTITY="1"
$env:USER_COUNT="100"
$env:REQUESTS="500"
$env:CONCURRENCY="50"
node .\tests\seckill\load-test.mjs
```

也可以直接用命令行参数覆盖：

```powershell
node .\tests\seckill\load-test.mjs --mode load --ticket-id 21 --requests 100 --concurrency 100
```

支持的 `--mode`：

| mode | 用途 |
| --- | --- |
| `load` | 常规并发下单压测 |
| `limit` | 单用户跨票档限购验证，默认使用 `--ticket-ids 21,22,23` |
| `soldout` | 多轮压测直到售罄，并追加售罄后探测 |
| `mq-failover` | 自动停 RabbitMQ、下单、观察 PEL、恢复 RabbitMQ |
| `cancel-rollback` | 创建待支付订单并取消，等待本地消息表驱动 Redis 回滚 |

重置测试环境：

```powershell
powershell -ExecutionPolicy Bypass -File .\tests\seckill\reset-test-env.ps1
```

自动对账：

```powershell
node .\tests\seckill\verify.mjs --ticket-id 21 --event-id 8
```

最近一次完整执行结果保存在 `tests/seckill/results/20260709-215810/summary.md`，`tests/seckill/results/LATEST.txt` 指向最新结果目录。

脚本行为：

1. 登录 `USER_COUNT` 个测试用户。
2. 按 `CONCURRENCY` 并发请求 `/order/create`。
3. 输出业务成功数、失败原因分布、HTTP 状态、延迟分位数。

## 4. 核心测试用例

### TC-01 高并发抢同一票档

目的：验证零超卖和 Redis 原子扣减。

执行：

```powershell
$env:TICKET_ID="21"
$env:QUANTITY="1"
$env:USER_COUNT="100"
$env:REQUESTS="1000"
$env:CONCURRENCY="100"
node .\tests\seckill\load-test.mjs
```

验收：

- 成功订单数不超过 `ticket_category.total_quantity`。
- MySQL `remaining_quantity >= 0`。
- Redis `ticket:stock:21 >= 0`。
- `order` 表同一票档订单数量不超过初始库存。
- 多余请求返回售罄、限流、限购或火爆提示。

对账：

```powershell
& "D:\JavaDevelop\mysql-8.0.28-winx64\bin\mysql.exe" -u root -p123456 -D school_ticket -e "
SELECT ticket_id,total_quantity,remaining_quantity FROM ticket_category WHERE ticket_id=21;
SELECT ticket_id, COUNT(*) order_count, SUM(quantity) sold_qty
FROM \`order\`
WHERE ticket_id=21 AND status IN (0,1,4)
GROUP BY ticket_id;
"
docker exec redis redis-cli GET ticket:stock:21
```

### TC-02 单用户重复下单

目的：验证用户级限流和活动级限购。

执行：

```powershell
$env:TICKET_ID="21"
$env:QUANTITY="1"
$env:USER_COUNT="1"
$env:REQUESTS="50"
$env:CONCURRENCY="20"
node .\tests\seckill\load-test.mjs
```

验收：

- 大部分请求被用户级限流拦截。
- 即使间隔执行多轮，同一活动累计购买数不超过 5。
- Redis `event:purchase:{eventId}` 中该用户数量不超过 5。

对账：

```powershell
docker exec redis redis-cli HGET event:purchase:8 1
& "D:\JavaDevelop\mysql-8.0.28-winx64\bin\mysql.exe" -u root -p123456 -D school_ticket -e "
SELECT user_id, SUM(quantity) qty
FROM \`order\`
WHERE user_id=1 AND ticket_id IN (SELECT ticket_id FROM ticket_category WHERE event_id=8)
  AND status IN (0,1,4)
GROUP BY user_id;
"
```

### TC-03 跨票档限购

目的：验证同一活动多票档合计最多 5 张。

前提：选择同一活动下多个票档，例如 event_id=8 的 `ticket_id=21,22,23`。

执行：

```powershell
$env:USER_COUNT="1"; $env:REQUESTS="3"; $env:CONCURRENCY="1"; $env:QUANTITY="2"; $env:TICKET_ID="21"; node .\tests\seckill\load-test.mjs
$env:USER_COUNT="1"; $env:REQUESTS="3"; $env:CONCURRENCY="1"; $env:QUANTITY="2"; $env:TICKET_ID="22"; node .\tests\seckill\load-test.mjs
$env:USER_COUNT="1"; $env:REQUESTS="3"; $env:CONCURRENCY="1"; $env:QUANTITY="2"; $env:TICKET_ID="23"; node .\tests\seckill\load-test.mjs
```

验收：

- 同一用户 event_id=8 下成功累计不超过 5 张。
- 超过 5 张的请求返回限购提示。
- Redis `event:purchase:8` 与 MySQL 有效订单数量一致。

### TC-04 售罄后无效流量过滤

目的：验证 Caffeine 本地售罄缓存 + Redis soldout 标记能快速失败。

执行：

1. 先用 TC-01 把票档抢空。
2. 继续打售罄后的请求。

```powershell
$env:TICKET_ID="21"
$env:USER_COUNT="100"
$env:REQUESTS="1000"
$env:CONCURRENCY="100"
node .\tests\seckill\load-test.mjs
```

验收：

- 成功订单数不再增加。
- 请求主要返回售罄或火爆。
- Redis `ticket:soldout:21` 存在。
- MySQL `order` 增量为 0。

辅助观察：

```powershell

& "D:\JavaDevelop\mysql-8.0.28-winx64\bin\mysql.exe" -u root -p123456 -D school_ticket -e "
SELECT COUNT(*) FROM \`order\` WHERE ticket_id=21;
"
```

### TC-05 Stream 到 RabbitMQ 投递可靠性

目的：验证 Lua 成功写 Stream 后，RabbitMQ 短暂不可用不会导致订单永久丢失。

执行：

1. 停 RabbitMQ。

```powershell
docker stop rabbitmq
```

2. 发起少量下单。

```powershell
$env:TICKET_ID="21"
$env:USER_COUNT="20"
$env:REQUESTS="20"
$env:CONCURRENCY="20"
node .\tests\seckill\load-test.mjs
```

3. 观察 Stream PEL 或消息未 ack。

```powershell
docker exec redis redis-cli XPENDING stream:orders order-consumers
```

4. 恢复 RabbitMQ。

```powershell
docker start rabbitmq
```

验收：

- RabbitMQ 恢复后 PEL 兜底重投。
- 订单最终落库。
- 同一 `order_no` 不重复落库。

注意：如果后端 RabbitMQ 连接未自动恢复，可重启后端再观察 PEL 补偿。

### TC-06 重复消息幂等

目的：验证重复消费不会重复扣 MySQL 库存。

执行方式：

1. 从 `stream:orders` 或后端日志中取一个已成功订单号。
2. 手工向 `order.create.queue` 重复投递同样 `orderId` 消息，或通过临时测试工具复投。

验收：

- `order` 表同一 `order_no` 只有一条记录。
- `ticket_category.remaining_quantity` 不会重复扣减。
- 后端日志出现“订单已存在，幂等跳过”。

### TC-07 支付与延时关单并发（本轮不执行）

目的：验证状态机 CAS。

执行：

1. 创建订单，取得 `orderNo`。
2. 临近过期时并发触发支付和关单。开发环境可直接调用支付接口，并手动把订单 `expire_time` 改到过去触发关单。

```powershell
& "D:\JavaDevelop\mysql-8.0.28-winx64\bin\mysql.exe" -u root -p123456 -D school_ticket -e "
UPDATE \`order\` SET expire_time = DATE_SUB(NOW(), INTERVAL 1 MINUTE) WHERE order_no='替换为订单号';
"
```

3. 同时请求：

```powershell
curl.exe -X POST http://localhost:8080/api/v1/order/pay -H "Content-Type: application/json" -H "Authorization: Bearer 替换token" -d "{\"orderNo\":\"替换为订单号\"}"
```

验收：

- 支付和关单只有一个成功。
- 如果关单成功，库存只回补一次。
- 如果支付成功，关单消费者跳过。
- 订单最终状态只能是 `1=已支付` 或 `2=已取消`，不能出现库存重复回补。

### TC-08 主动取消回滚

目的：验证 MySQL 回补 + 本地消息表驱动 Redis 回滚。

执行：

1. 创建一个订单。
2. 调 `/order/cancel`。
3. 等待 10 秒以上，让 `OrderEventLogTask` 执行。

验收：

- 订单状态为 `2`。
- MySQL 票档库存增加。
- `order_event_log` 先写入，最终 `status=1`。
- Redis `ticket:stock:{ticketId}` 回补。
- Redis `event:purchase:{eventId}` 中用户购买数减少。

对账：

```powershell
& "D:\JavaDevelop\mysql-8.0.28-winx64\bin\mysql.exe" -u root -p123456 -D school_ticket -e "
SELECT order_no,status FROM \`order\` WHERE order_no='替换为订单号';
SELECT * FROM order_event_log WHERE order_no='替换为订单号';
"
docker exec redis redis-cli GET ticket:stock:21
docker exec redis redis-cli HGETALL event:purchase:8
```

### TC-09 退款幂等（本轮不执行）

目的：验证退款表一单一退和回补幂等。

执行：

1. 创建订单并支付。
2. 并发多次调用 `/order/refund`。

验收：

- `refund` 表同一 `refund_id=orderNo` 只有一条。
- 订单最终状态为 `3=已退款`。
- MySQL 库存只回补一次。
- `order_event_log` 退款事件只生效一次。

注意：当前项目没有真实三方支付网关，因此本用例验证的是本地退款幂等与库存回补，不验证外部支付一致性。

### TC-10 Redis 回滚失败补偿

目的：验证取消/退款/超时链路在 Redis 暂不可用时不会丢补偿事件。

执行：

1. 创建待支付订单。
2. 停 Redis。

```powershell
docker stop redis
```

3. 调用取消或退款。
4. 恢复 Redis。

```powershell
docker start redis
```

验收：

- MySQL 状态和库存变更已完成。
- `order_event_log` 保留待处理或失败记录。
- Redis 恢复后，任务重试并最终把 `order_event_log.status` 置为 `1`。
- 若达到最大重试次数，记录为 `2=失败`，需要人工补偿。

## 5. 最终一致性对账

压测或异常测试结束后执行以下对账。

### 5.1 MySQL 库存对账

```sql
SELECT
  tc.ticket_id,
  tc.total_quantity,
  tc.remaining_quantity,
  COALESCE(SUM(CASE WHEN o.status IN (0,1,4) THEN o.quantity ELSE 0 END), 0) AS occupied_qty,
  tc.remaining_quantity + COALESCE(SUM(CASE WHEN o.status IN (0,1,4) THEN o.quantity ELSE 0 END), 0) AS calculated_total
FROM ticket_category tc
LEFT JOIN `order` o ON o.ticket_id = tc.ticket_id
WHERE tc.ticket_id = 21
GROUP BY tc.ticket_id, tc.total_quantity, tc.remaining_quantity;
```

通过标准：

- `calculated_total = total_quantity`。
- `remaining_quantity >= 0`。

### 5.2 Redis 与 MySQL 对账

```powershell
docker exec redis redis-cli GET ticket:stock:21
docker exec redis redis-cli HGETALL event:purchase:8
```

通过标准：

- 在异步任务完成后，Redis 库存与 MySQL 可用库存口径一致。
- `event:purchase:{eventId}` 中用户数量与有效订单占用一致。

### 5.3 本地消息表对账

```sql
SELECT status, COUNT(*) FROM order_event_log GROUP BY status;
SELECT status, COUNT(*) FROM refund GROUP BY status;
```

通过标准：

- 正常情况下不长期存在 `status=0` 或 `status=9`。
- `status=2` 需要人工确认原因，不能静默忽略。

## 6. 通过标准汇总

| 能力 | 通过标准 |
| --- | --- |
| 零超卖 | 成功订单占用数量不超过总库存，Redis/MySQL 库存不为负 |
| 限购 | 同一用户同一活动跨票档累计不超过 5 张 |
| 削峰 | 售罄后订单数不再增加，无效请求快速失败 |
| 可靠投递 | RabbitMQ 短暂异常后，Stream PEL 可重投，订单最终落库 |
| 消费幂等 | 重复消息不重复生成订单、不重复扣 MySQL 库存 |
| 状态机 | 取消、超时等非支付状态流转只能有一个合法结果 |
| 逆向补偿 | 取消/超时后 MySQL 和 Redis 库存最终一致 |
| 退款幂等 | 本轮不执行，留作支付接入后的扩展项 |

## 7. 当前项目边界

当前项目是模拟支付，不包含真实支付网关，因此以下内容不能写成“已实现”：

- 支付网关回调乱序处理。
- 关单前主动查证三方支付真实状态。
- 先关单后支付回调触发自动退款。
- 退款指数退避。
- 未知退款状态主动查单对齐。
- 告警系统。

这些可以作为后续接入真实支付时的扩展测试项。
